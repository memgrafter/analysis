# The KnowWhereGraph Ontology

## Quick Facts
- arXiv ID: 2410.13948
- Source URL: https://arxiv.org/abs/2410.13948
- Reference count: 40
- Integrates 30+ geospatial datasets (28 billion triples) using modular ontology modeling

## Executive Summary
This paper presents the KnowWhereGraph Ontology, a large-scale geospatial knowledge graph developed using modular ontology modeling to integrate over 30 heterogeneous datasets including hazards, climate variables, soil properties, demographics, and health data. The ontology provides a schema enabling rapid area briefings for decision-makers and data scientists, answering questions like "What is here?" and "What happened here before?" through efficient spatial integration and semantic harmonization. Key innovations include leveraging a discrete global grid (S2) as a spatial backbone, extracting a reusable SOSA/SSN kernel for consistent observation modeling, and developing novel ontology design patterns for hierarchical features, causal relations, and taxonomy alignment.

## Method Summary
The KnowWhereGraph Ontology was developed using a modular ontology modeling (MOMo) approach that combines top-down pattern extraction with bottom-up dataset integration. The methodology centers on extracting a reusable SOSA/SSN kernel for consistent observation modeling across diverse datasets, while using a discrete global grid (S2) as a spatial backbone to enable efficient integration of both vector and raster geospatial data. Novel ontology design patterns were created for hierarchical features, causal relations, and taxonomy alignment to support the integration of 30+ heterogeneous datasets totaling 28 billion triples. The approach maintains a tripartite disconnect between observations, features of interest, and observed properties to enable semantic harmonization while supporting rapid area briefings for humanitarian response and decision-making use cases.

## Key Results
- Successfully integrated 30+ geospatial datasets (28 billion triples) using modular ontology modeling
- Leveraged discrete global grid (S2) as spatial backbone for efficient spatial relationship pre-computation
- Extracted reusable SOSA/SSN kernel enabling semantic harmonization across heterogeneous observation data
- Developed novel ontology design patterns for hierarchical features, causal relations, and taxonomy alignment
- Enabled rapid area briefings answering "What is here?", "What happened here before?", and "How does it compare to other regions?"

## Why This Works (Mechanism)

### Mechanism 1
- Claim: Using a Discrete Global Grid (DGG) as the spatial backbone enables efficient geospatial integration across heterogeneous datasets.
- Mechanism: The DGG provides a common spatial reference system that allows pre-computation of spatial relationships and summary statistics, enabling rapid area briefings without recalculating spatial joins.
- Core assumption: All geospatial phenomena can be meaningfully mapped onto a hierarchical grid structure without significant loss of precision for the use cases.
- Evidence anchors:
  - [abstract] "The ontology provides a schema enabling rapid area briefings for decision-makers and data scientists, answering questions like 'What is here?' and 'What happened here before?'"
  - [section 2.1] "A DGG allows for a common underlying spatial reference system for both geospatial vector and raster datasets, for which we can pre-compute spatial relations between different features or regions"
  - [corpus] Weak evidence - corpus doesn't provide direct support for DGG integration mechanisms
- Break condition: When spatial phenomena are too granular or irregularly shaped for meaningful grid cell representation, or when the DGG resolution is too coarse for required precision.

### Mechanism 2
- Claim: The modular ontology modeling (MOMo) approach enables rapid integration of new datasets while maintaining schema consistency.
- Mechanism: By extracting reusable patterns from existing ontologies (particularly SOSA/SSN kernel) and instantiating them for specific use cases, the ontology can quickly incorporate new data sources with predictable structure.
- Core assumption: Key notions and competency questions can be identified early enough to guide pattern selection and instantiation.
- Evidence anchors:
  - [section 3.1.5] "This step is the process by which a general or abstract pattern is taken and modified to fit the use case, creating modules through template-based instantiation"
  - [abstract] "Key contributions include leveraging a discrete global grid (S2) as a spatial backbone, extracting a reusable kernel from the SOSA/SSN ontology for consistent observation modeling"
  - [corpus] Weak evidence - corpus doesn't provide examples of modular ontology modeling success/failure
- Break condition: When dataset characteristics are too heterogeneous for existing patterns to capture, or when domain concepts evolve faster than pattern adaptation can accommodate.

### Mechanism 3
- Claim: The observation-centric modeling approach (using SOSA/SSN kernel) enables semantic harmonization across diverse datasets describing the same phenomena.
- Mechanism: By maintaining a tripartite disconnect between observations, features of interest, and observed properties, the ontology can integrate datasets that measure different aspects of the same phenomena without requiring complete schema alignment.
- Core assumption: Different datasets can be meaningfully related through their spatiotemporal dimensions even when their observable properties differ significantly.
- Evidence anchors:
  - [section 4.4.3] "By maintaining a tripartite purposeful disconnect between an observation, the target of the observation (i.e., the feature of interest) and the observed property, allows for quick semantic harmonization between terms"
  - [abstract] "developing novel ontology design patterns for hierarchical features, causal relations, and taxonomy alignment"
  - [corpus] Weak evidence - corpus doesn't provide direct evidence of observation-centric modeling success
- Break condition: When datasets lack sufficient spatiotemporal metadata to enable meaningful alignment, or when observed properties are fundamentally incompatible.

## Architecture Onboarding

- Component map: KWG Core (Hazard, Region, Cell classes) -> SOSA/SSN kernel for observations -> Integrated datasets mapped through these schemas -> Metadata management -> Spatial relations (GeoSPARQL) -> Time modeling (OWL-Time) -> Quantity representation (QUDT)

- Critical path: Raw datasets → Spatial alignment to S2 cells → Observation modeling using SOSA/SSN patterns → Metadata annotation → Queryable knowledge graph

- Design tradeoffs: Using hierarchical DGG provides computational efficiency but may lose precision for very local phenomena; modular approach enables rapid integration but requires careful pattern selection; observation-centric model enables semantic harmonization but increases query complexity

- Failure signatures: Integration failures manifest as missing spatial relationships, inconsistent observation properties, or inability to answer competency questions; metadata issues show up as provenance gaps or ambiguous dataset boundaries

- First 3 experiments:
  1. Test spatial integration by querying for all hazards within a specific S2 cell and verifying spatial relationships with adjacent cells
  2. Test observation modeling by integrating a simple dataset (e.g., temperature observations) and verifying the SOSA/SSN pattern instantiation
  3. Test metadata functionality by querying for dataset provenance and verifying the subgraph relationships between thematic and dataset subgraphs
