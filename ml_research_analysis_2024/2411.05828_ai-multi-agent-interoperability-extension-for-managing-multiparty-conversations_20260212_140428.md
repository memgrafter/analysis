# AI Multi-Agent Interoperability Extension for Managing Multiparty Conversations

## Quick Facts
- arXiv ID: 2411.05828
- Source URL: https://arxiv.org/abs/2411.05828
- Reference count: 25
- Primary result: Introduces OVON framework extension for multiparty AI conversations with Floor Manager, Convener Agent, and shared Floor concept

## Executive Summary
This paper extends the OVON framework to enable multiparty AI conversations by introducing a structured approach to agent coordination and context sharing. The key innovation is the introduction of the Floor as a shared conversational space, managed by a Floor Manager for message routing and a Convener Agent for orchestrating agent participation. These enhancements allow agents developed with different technologies to collaborate through natural language-based APIs, improving scalability, security, and efficiency in managing complex, dynamic interactions among multiple AI agents.

## Method Summary
The paper proposes an extension to the OVON framework by introducing three key components: the Floor as a shared conversational space for context management, the Floor Manager for neutral message routing and communication control, and the Convener Agent for orchestrating agent participation and managing conversational dynamics. The method involves implementing standardized JSON message envelopes, defining roles for each component, and establishing protocols for agent invitation, Floor requests, and context sharing. The approach builds upon existing OVON specifications while adding multiparty conversation capabilities.

## Key Results
- Introduction of the Floor concept as a shared conversational space enabling context management and collaborative knowledge building
- Implementation of the Floor Manager for message routing and communication control, ensuring orderly message flow
- Development of the Convener Agent for agent orchestration, managing turn-taking and participant invitations

## Why This Works (Mechanism)

### Mechanism 1
- Claim: The Floor Manager enforces orderly message flow by routing and controlling visibility, preventing message chaos in multiparty interactions.
- Mechanism: The Floor Manager acts as a neutral message router that determines which messages are sent to which participants, ensuring that only authorized messages reach the intended recipients. It maintains participant lists and enforces policies about who has the Floor at any given time.
- Core assumption: The Floor Manager operates without agency, meaning it doesn't make autonomous decisions beyond message routing, which ensures neutrality and prevents bias in message distribution.
- Evidence anchors:

  - [section] "The Floor Manager controls the Floor. It does not present itself as a direct actor in the conversation and restricts its agency to facilitating communication. It is a trusted neutral actor."

- Break condition: If the Floor Manager is compromised or becomes an active agent with decision-making capabilities, it could introduce bias or security vulnerabilities in message routing.

### Mechanism 2
- Claim: The Convener Agent coordinates conversational dynamics by managing turn-taking and participant invitations, enabling structured multiparty dialogue.
- Mechanism: The Convener Agent serves as the central coordinator that processes Floor requests from agents, evaluates them based on predefined rules, and grants or revokes Floor privileges to maintain orderly communication. It maintains a queue to manage simultaneous requests and prioritizes them based on urgency and context.
- Core assumption: The Convener Agent can effectively evaluate Floor requests and make fair decisions about granting speaking privileges without introducing bias or inefficiency.
- Evidence anchors:

- Break condition: If the Convener Agent becomes overwhelmed with too many simultaneous requests or fails to properly evaluate priorities, it could create bottlenecks or unfair speaking opportunities.

### Mechanism 3
- Claim: The unified Floor concept provides a shared conversational space that enables context management and collaborative knowledge building across agents.
- Mechanism: The Floor acts as a shared conversational space that can maintain a unified context for all participants, allowing agents to access and contribute to the collective knowledge base. This centralized approach enables agents to reference shared information without redundancy and build upon each other's contributions.
- Core assumption: Centralized context management through the Floor is more efficient and effective than distributed context management where each agent manages its own context.
- Evidence anchors:

## Architecture Onboarding

- Component map:
  - Floor Manager -> Message routing and communication control
  - Convener Agent -> Conversational coordination and turn management
  - Floor -> Shared conversational space and context repository
  - Individual Agents -> Participants with specific capabilities and knowledge domains
  - Message Envelope -> Standardized communication format (OVON specifications)

- Critical path: User request → Convener Agent initiates conversation → Floor Manager sets up message routing → Agents join via invitation → Conversation proceeds with turn-taking → Context updates flow through Floor → Conversation ends or agents leave

- Design tradeoffs:
  - Centralized vs. decentralized context management: Centralized provides consistency but creates single point of failure; decentralized provides resilience but may cause context fragmentation
  - Message routing complexity vs. performance: More sophisticated routing rules improve control but increase processing overhead
  - Invitation control strictness vs. flexibility: Strict control improves security but may reduce system adaptability

- Failure signatures:
  - Message routing failures: Agents not receiving messages, duplicate messages, or messages going to wrong recipients
  - Context synchronization issues: Agents operating on outdated information, context conflicts, or inconsistent state
  - Invitation/authentication failures: Unauthorized agents gaining access or legitimate agents being blocked

- First 3 experiments:
  1. Implement basic message routing between two agents using the Floor Manager, verifying that messages flow correctly and only reach intended recipients
  2. Add the Convener Agent to coordinate turn-taking between three agents, testing Floor request and grant mechanisms
  3. Implement shared context management through the Floor, testing context updates and access by multiple agents simultaneously
